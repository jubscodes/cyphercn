## Pattern Analysis

**Current MDX Structure:**
```mdx
---
title: Button
description: Displays an 8-bit button component.
---

import { Button } from "@/components/ui/8bit/button";
import CopyCommandButton from "@/components/copy-command-button";

<CopyCommandButton
  copyCommand="pnpm dlx shadcn@latest add @8bitcn/button"
  command="pnpm dlx shadcn@latest add @8bitcn/button"
/>

<ComponentPreview title="8-bit button component" name="button">
```

**Target Structure:**
```mdx
---
title: Button
description: Displays a CyberCN button component.
---

import { Button } from "@/components/ui/8bit/button";
import CopyCommandButton from "@/components/copy-command-button";

<CopyCommandButton
  copyCommand="pnpm dlx shadcn@latest add @cybercn/button"
  command="pnpm dlx shadcn@latest add @cybercn/button"
/>

<ComponentPreview title="CyberCN button component" name="button">
```

Note: Import paths stay `@/components/ui/8bit/` - only user-facing text changes.

## Execution Strategy

1. **Wave 1**: Update shared components (installation-commands, index.mdx) + theme system
2. **Wave 2**: Batch update all component MDX files (sed-replaceable patterns)
3. **Wave 3**: Batch update all block MDX files

## Risks

- **Registry not configured**: The `@cybercn/` registry URL needs to exist for installs to work (Phase 4 handles this)
- **Import paths unchanged**: Keep `@/components/ui/8bit/` paths - changing them would break everything

## Decision

Keep docs clean like shadcn. Only update branding text, not structural changes to the documentation site itself.
