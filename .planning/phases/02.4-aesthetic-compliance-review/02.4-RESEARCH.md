# Phase 2.4: Aesthetic Compliance Review - Research

**Researched:** 2026-02-05
**Domain:** Component library design consistency audit and ASCII art generation
**Confidence:** HIGH

## Summary

Phase 2.4 focuses on reviewing all existing cypher components to ensure compliance with the MS-DOS terminal aesthetic established in prior phases. Research identified three primary audit areas: (1) visual element compliance (replacing pixelated graphics with ASCII art alternatives), (2) copy/terminology consistency (ensuring terminal-style language throughout), and (3) CSS class and styling verification (checking proper use of .cyphercn and .cyphercn-normal classes).

The established aesthetic standards from STATE.md require thin 1px borders instead of pixel blocks, IBM Plex Mono font, monochrome themes only, terminal terminology (not 8-bit/pixel/retro language), box-drawing characters for borders, and @cyphercn/ namespace usage. Current codebase examination revealed several non-compliance issues: the avatar.tsx component uses a complex pixelated frame (lines 41-98) built with absolute-positioned divs, multiple blocks contain "retro" className usage (8 files identified), main-menu.tsx includes "Retro 8-bit Quest" copy, and game-over.tsx references "/images/8-bit-skull.png".

Industry best practices for component consistency audits emphasize systematic review of all states (default, hover, disabled), adherence to design system standards, and automated tooling where possible. ASCII art generation for terminal UIs should use established box-drawing characters from Unicode block U+2500 to U+257F, leverage FIGlet-style tools for text banners, and maintain proper aspect ratio (terminal characters are ~2:1 height-to-width ratio).

**Primary recommendation:** Conduct systematic component-by-component review using a checklist-based approach, replace pixelated visual elements with ASCII/box-drawing alternatives, update all copy to terminal-style language, and verify CSS class usage matches established patterns.

## Architecture Patterns

### Recommended Review Structure

**Component Audit Order:**
```
1. Core components (button, card, input, etc.) - verify base patterns
2. Complex components (avatar, empty, icons) - identify visual replacements
3. Block components (main-menu, game-over, etc.) - verify copy compliance
4. Consistency check - verify all use .cyphercn/.cyphercn-normal correctly
```

### Pattern 1: Systematic Component Review Checklist

**What:** Structured checklist applied to each component ensuring all aesthetic criteria are met

**When to use:** For every .tsx file in components/ui/cypher/ and components/ui/cypher/blocks/

**Checklist items:**
```markdown
Per component:
- [ ] No "retro", "8-bit", "8bit", "pixel", or non-terminal terminology in copy
- [ ] CSS classes use .cyphercn (uppercase) or .cyphercn-normal (normal case)
- [ ] No pixelated image frames or SVG-based decorative elements
- [ ] Borders use thin 1px or box-drawing characters, not pixel blocks
- [ ] Text follows terminal convention (uppercase labels, normal case descriptions)
- [ ] Font usage matches established pattern (IBM Plex Mono via font-mono)
- [ ] Props follow CyphercnXProps naming convention
- [ ] No rounded corners (rounded-none or sharp edges only)
```

### Pattern 2: ASCII Art Replacement Strategy

**What:** Replace pixelated visual elements with ASCII art or box-drawing character equivalents

**When to use:** For decorative frames, icons, and visual separators currently using pixel-based graphics

**Implementation approach:**
```tsx
// BEFORE: Pixelated div-based frame (avatar.tsx lines 41-98)
<div className="absolute inset-0 w-full h-full pointer-events-none">
  <div className="absolute top-0 left-[23%] right-[23%] h-[7%] bg-foreground"></div>
  {/* ...96 more lines of positioned divs */}
</div>

// AFTER: ASCII box-drawing frame
<div className="absolute inset-0 pointer-events-none box-drawing">
  ┌─────────┐
  │         │
  │         │
  └─────────┘
</div>

// Or simpler: use existing border patterns
<div className="border border-foreground">
  {/* content */}
</div>
```

### Pattern 3: Terminology Standardization

**What:** Replace non-terminal language with command-center appropriate terms

**When to use:** All user-facing copy in components and blocks

**Translation guide:**
```typescript
// Non-compliant → Terminal-style
"Retro 8-bit Quest"     → "COMMAND CENTER" or "TERMINAL INTERFACE"
"retro" className       → "cyphercn" or "cyphercn-normal"
"Pixel-perfect"         → "Precision rendering"
"8-bit graphics"        → "Terminal graphics"
"Pixelated"             → "Monochrome" or "Terminal-rendered"
```

### Anti-Patterns to Avoid

- **Automated bulk find/replace:** Terminology changes require context awareness; automated replacement may break component logic
- **Removing all decorative elements:** ASCII art replacements should maintain visual hierarchy and aesthetic appeal
- **Inconsistent ASCII styles:** If using box-drawing, stick to single or double line consistently (don't mix randomly)

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ASCII art generation | Custom pixel-to-ASCII converter | FIGlet for text, manual Unicode box-drawing for frames | FIGlet has 500+ fonts optimized for terminals; Unicode box-drawing characters (U+2500-257F) are standard |
| Pattern searching in codebase | Manual file-by-file review | grep/ripgrep with regex patterns | Automated search ensures no instances missed; ripgrep respects .gitignore |
| Component variant validation | Manual prop checking | Existing TypeScript + CVA infrastructure | Type system already enforces variant consistency |
| Visual consistency checking | Screenshot comparison tools | Manual review with checklist | One-time audit doesn't justify tooling setup; checklist ensures systematic coverage |

**Key insight:** Audit tooling should leverage existing codebase infrastructure (TypeScript, ESLint) rather than introducing new dependencies for a one-time review.

## Common Pitfalls

### Pitfall 1: Incomplete Terminology Search

**What goes wrong:** Searching only for obvious terms like "8-bit" misses variants like "8bit", "retro", className="pixel", or image file names

**Why it happens:** Non-terminal language appears in multiple contexts: component copy, CSS classes, prop names, file paths, comments

**How to avoid:** Use comprehensive search patterns covering all contexts:
```bash
# Search component text content
grep -r "8-bit\|8bit\|pixel\|retro\|Retro" components/ui/cypher/ --include="*.tsx"

# Search CSS classes specifically
grep -r 'className.*retro\|className.*pixel' components/ui/cypher/ --include="*.tsx"

# Search image references
grep -r '\.png\|\.jpg\|\.svg' components/ui/cypher/ --include="*.tsx"

# Search comments (may contain outdated terminology)
grep -r '//.*retro\|//.*8-bit' components/ui/cypher/ --include="*.tsx"
```

**Warning signs:** Finding instances through manual browsing that weren't caught by initial searches

### Pitfall 2: Breaking Existing Component Functionality

**What goes wrong:** Removing the pixelated avatar frame breaks layout assumptions in consuming components; changing className from "retro" to "cyphercn" breaks CSS selectors

**Why it happens:** Components have implicit dependencies on specific DOM structure or CSS class names

**How to avoid:**
1. Check component usage before major structural changes: `grep -r "Avatar.*variant" components/`
2. Verify CSS dependencies: check if removed classNames are referenced in stylesheets
3. Test in isolation: render component before/after changes to verify visual parity
4. Maintain backward compatibility: keep variant options, just change defaults

**Warning signs:** Components that look correct in isolation but break in block compositions

### Pitfall 3: Inconsistent ASCII Art Style

**What goes wrong:** Using single-line box drawing (─) in some components but double-line (═) in others creates visual inconsistency

**Why it happens:** Different developers/phases use different Unicode characters without style guide

**How to avoid:** Establish ASCII art style guide at start of audit:
- Frame borders: single-line (─ │ ┌ ┐ └ ┘) for default, double (═ ║ ╔ ╗ ╚ ╝) for emphasis
- Icons: prefer simple geometric shapes over complex ASCII art
- Text banners: use FIGlet with consistent font choice (e.g., "standard" or "banner")

**Warning signs:** Components side-by-side using visually different border characters for same semantic purpose

### Pitfall 4: Over-Aggressive Removal of Visual Elements

**What goes wrong:** Removing avatar's pixelated frame entirely leaves component looking plain; matches "terminal aesthetic" literally but loses visual interest

**Why it happens:** Interpreting "thin borders instead of pixel blocks" as "no decorative elements"

**How to avoid:** Terminal aesthetic includes visual richness through ASCII art, box-drawing, and scanlines. Replace, don't remove:
- Pixelated frame → ASCII box-drawing frame or thin double border
- Decorative icons → Simple geometric ASCII equivalents
- Image placeholders → ASCII art representations

**Warning signs:** Components that feel visually flat or indistinguishable from each other

### Pitfall 5: Missing Documentation/Block MDX Files

**What goes wrong:** Updating component code but forgetting to update MDX documentation files that contain outdated copy

**Why it happens:** Documentation lives in separate directory (content/docs/) from components

**How to avoid:** For each component updated, check for corresponding MDX:
```bash
# Find documentation for a component
find content/docs -name "component-name.mdx"

# Review description field in frontmatter
# Review example code in MDX
```

**Warning signs:** Component looks correct but documentation page shows old terminology
