# Phase 2.3: Cyberpunk Stylesheet - Research

**Researched:** 2026-02-04
**Domain:** CSS cyberpunk/terminal aesthetic styling
**Confidence:** HIGH

## Summary

This research investigates techniques for implementing a cyberpunk/terminal aesthetic stylesheet using CSS-only approaches for phosphor glow effects, CRT scanlines, and monochrome color theming. The technical domain centers on layered text-shadow patterns for glow effects, linear gradient overlays for scanlines, OKLCH color space for perceptually uniform brightness, and IBM Plex Mono typography via next/font/google.

The standard approach is **pure CSS with no JavaScript dependencies** for all core effects. Dithering capabilities exist but are out of scope for the stylesheet itself—they require Canvas-based JavaScript libraries and are better suited for image preprocessing or opt-in component features rather than global CSS utilities.

Box-drawing characters (┌─┐│└┘ etc.) are standard Unicode characters requiring no library—they work directly in JSX/HTML with proper UTF-8 encoding and monospace fonts.

**Primary recommendation:** Build cyberpunk.css using CSS custom properties with currentColor fallback, implement phosphor glow with 5-7 layered text-shadows (white base + colored glow), create scanlines with linear-gradient pseudo-elements using pointer-events: none, and use OKLCH color space for all monochrome variants to ensure perceptually uniform brightness.

### Pattern 1: CSS Custom Properties with currentColor Fallback
**What:** Define phosphor glow colors using CSS variables that fallback to currentColor for automatic theme awareness
**When to use:** All color-dependent utilities (phosphor-glow, phosphor-border-glow)
**Example:**
```css
/* Source: MDN CSS Custom Properties + research synthesis */
.phosphor-glow {
  text-shadow:
    /* White foundation layer (small blur) */
    0 0 7px #fff,
    0 0 10px #fff,
    0 0 21px #fff,
    /* Colored glow layer (large blur) - uses custom property or currentColor */
    0 0 42px var(--glow-color, currentColor),
    0 0 82px var(--glow-color, currentColor),
    0 0 102px var(--glow-color, currentColor),
    0 0 151px var(--glow-color, currentColor);
}

.phosphor-border-glow {
  box-shadow:
    0 0 5px var(--glow-color, currentColor),
    0 0 10px var(--glow-color, currentColor),
    inset 0 0 10px var(--glow-color, currentColor);
  border: 1px solid currentColor;
}
```

### Pattern 2: Scanline Overlay with Pseudo-Elements
**What:** Use ::before pseudo-element with linear gradients and pointer-events: none for non-blocking scanline effects
**When to use:** CRT aesthetic overlays that must not interfere with user interaction
**Example:**
```css
/* Source: dev.to CRT terminal tutorial + aleclownes.com synthesis */
.crt-scanlines::before {
  content: "";
  display: block;
  position: absolute;
  top: 0;
  left: 0;
  bottom: 0;
  right: 0;
  background: linear-gradient(
    to bottom,
    rgba(18, 16, 16, 0) 50%,
    rgba(0, 0, 0, 0.25) 50%
  );
  background-size: 100% 4px; /* Creates 2px transparent, 2px opaque pattern */
  z-index: 10;
  pointer-events: none; /* CRITICAL: Allows clicks to pass through */
}

.crt-scanlines-subtle::before {
  content: "";
  display: block;
  position: absolute;
  top: 0;
  left: 0;
  bottom: 0;
  right: 0;
  background: linear-gradient(
    to bottom,
    rgba(0, 0, 0, 0) 50%,
    rgba(0, 0, 0, 0.1) 50%
  );
  background-size: 100% 2px; /* Finer scanlines */
  z-index: 10;
  pointer-events: none;
}
```

### Pattern 5: Animation with prefers-reduced-motion Guard
**What:** Wrap all animations (glow pulsing, scanline movement) in @media (prefers-reduced-motion: no-preference)
**When to use:** ALL animations to ensure accessibility compliance (WCAG 2.3.3)
**Example:**
```css
/* Source: MDN prefers-reduced-motion + W3C WCAG C39 technique */
@media (prefers-reduced-motion: no-preference) {
  @keyframes phosphor-pulse {
    0%, 100% {
      text-shadow:
        0 0 7px #fff,
        0 0 10px #fff,
        0 0 21px #fff,
        0 0 42px var(--glow-color, currentColor),
        0 0 82px var(--glow-color, currentColor);
    }
    50% {
      text-shadow:
        0 0 5px #fff,
        0 0 7px #fff,
        0 0 15px #fff,
        0 0 30px var(--glow-color, currentColor),
        0 0 60px var(--glow-color, currentColor);

/* No animation for users who prefer reduced motion - static glow only */
```

### Pattern 6: Box-Drawing Characters (No Library)
**What:** Use Unicode box-drawing characters (U+2500–U+257F) directly in JSX/HTML
**When to use:** DOS-style panels, borders, ASCII art decorations
**Example:**
```tsx

  single: {
    tl: "┌", tr: "┐", bl: "└", br: "┘",
    h: "─", v: "│", lt: "├", rt: "┤",
  },

    tl: "╔", tr: "╗", bl: "╚", br: "╝",

// Use in JSX
<div className="cyphercn">
  {chars.single.tl}{chars.single.h.repeat(20)}{chars.single.tr}

```

### Anti-Patterns to Avoid
- **Random z-index values:** Don't use arbitrary numbers (999, 9999). Use a consistent scale defined in CSS variables (--z-scanlines: 10, --z-vignette: 20, etc.)
- **Blocking overlays:** Never create pointer-events-blocking overlays without pointer-events: none. This breaks interactivity.
- **Excessive text-shadow layers:** More than 7-8 shadows causes performance issues on lower-end devices. Stick to 5-7 for optimal balance.
- **HSL for monochrome variants:** HSL lightness is NOT perceptually uniform. Use OKLCH for consistent brightness.
- **Animations without motion guards:** Always wrap animations in @media (prefers-reduced-motion: no-preference) for accessibility.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Font optimization | Manual @font-face with preload | next/font/google | Automatic font subsetting, FOUT prevention, optimal loading strategy |
| Dithering algorithms | Custom pixel manipulation | canvas-dither or CanvasDither | Floyd-Steinberg, Atkinson, Bayer matrix are complex; existing implementations handle edge cases |
| Color space conversion | Manual RGB→OKLCH math | Native OKLCH() CSS function | Browser-native, hardware-accelerated, handles gamut mapping |
| Scanline patterns | Canvas-based rendering | CSS linear-gradient with background-size | CSS is declarative, performant, and doesn't require JavaScript |

**Key insight:** The cyberpunk aesthetic is achievable with pure CSS patterns—resist the urge to over-engineer with JavaScript or Canvas unless implementing advanced features like image dithering (which should be opt-in components, not global stylesheet utilities).

## Common Pitfalls

### Pitfall 1: Phosphor Glow Performance on Large Text Blocks
**What goes wrong:** Applying multi-layer text-shadow to large paragraphs causes visible lag on scroll/repaint
**Why it happens:** Text-shadow is expensive to render—each shadow layer multiplies the rendering cost
**How to avoid:** Use phosphor-glow sparingly on headings, labels, and short text. For body text, use normal text with colored foreground only. Consider a lighter 3-shadow variant for less critical elements.
**Warning signs:** Janky scrolling, slow repaint, high paint times in Chrome DevTools Performance panel

### Pitfall 2: Scanline Overlay Blocks Interactions
**What goes wrong:** Scanline pseudo-element prevents clicks/touches on underlying content
**Why it happens:** Forgot `pointer-events: none` on the ::before overlay
**How to avoid:** ALWAYS include `pointer-events: none` on decorative overlay pseudo-elements
**Warning signs:** Buttons don't click, links don't work, form inputs can't be focused

### Pitfall 3: OKLCH Browser Fallback Ignored
**What goes wrong:** Colors appear broken in older browsers (pre-2023)
**Why it happens:** OKLCH is Baseline Widely Available since May 2023, but older browsers need fallbacks
**How to avoid:** Provide RGB fallback before OKLCH declaration:
```css
color: rgb(0, 255, 136); /* Fallback */
color: oklch(65% 0.25 145); /* Modern browsers */
```
**Warning signs:** Color appears black/transparent in Safari < 15.4, Chrome < 111, Firefox < 113

### Pitfall 4: currentColor with Nested Components
**What goes wrong:** Glow color inherits unexpected colors from parent elements
**Why it happens:** currentColor resolves to the element's computed color value, which can inherit
**How to avoid:** Explicitly set --glow-color on themed containers, don't rely on implicit currentColor behavior in complex nesting
**Warning signs:** Glow colors change unexpectedly when component is used in different contexts

### Pitfall 5: z-index Stacking Context Issues
**What goes wrong:** Scanlines appear below content or above modals incorrectly
**Why it happens:** Each positioned element with z-index creates a new stacking context; scanline overlays must be positioned correctly within their context
**How to avoid:** Follow the established z-index scale from STATE.md (content < scanlines: 10 < vignette: 20 < reflection: 21). Never use z-index > 50 for scanlines.
**Warning signs:** Scanlines visible above modal overlays, or hidden below background images

### Pitfall 6: Letter-Spacing Breaks Box-Drawing Characters
**What goes wrong:** Box-drawing borders have gaps when letter-spacing is applied
**Why it happens:** Letter-spacing adds space between characters, breaking continuous border lines
**How to avoid:** Set `letter-spacing: 0` on box-drawing character containers, or use `white-space: nowrap; overflow: hidden` with sufficient repeats
**Warning signs:** Visible gaps in ┌──┐ style borders

### Pitfall 7: IBM Plex Mono Loading Delays
**What goes wrong:** Font flash (FOUT) or invisible text during load
**Why it happens:** next/font/google default behavior with slow network
**How to avoid:** Use `display: 'swap'` in font config for immediate fallback rendering. Consider preloading on critical pages.
**Warning signs:** Blank text on initial load, or flash from Courier New → IBM Plex Mono

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| HSL for color theming | OKLCH for perceptual uniformity | May 2023 (CSS Color 4/5) | Same L value = same perceived brightness (critical for monochrome variants) |
| @font-face with manual preload | next/font/google | Next.js 13+ (Oct 2022) | Automatic optimization, subsetting, FOUT prevention |
| JavaScript scanline rendering | CSS linear-gradient overlays | Established pattern (~2017+) | Zero JavaScript, better performance, simpler code |
| Inline style animations | @media prefers-reduced-motion guards | WCAG 2.1 (2018) / 2.3.3 | Accessibility compliance, respect user preferences |

**Deprecated/outdated:**
- **box-shadow on text for glow:** Use text-shadow instead—more efficient for text, better rendering quality
- **Canvas-based scanlines:** Use CSS gradients unless you need pixel-perfect CRT curvature/distortion effects
- **Hardcoded RGB colors:** Use OKLCH with RGB fallbacks for modern color management

## Open Questions

Things that couldn't be fully resolved:

1. **Dithering Library Choice for Future Image Features**
   - What we know: canvas-dither (1.0.3) and CanvasDither both support Atkinson/Floyd-Steinberg; both are actively used
   - What's unclear: Which has better performance for real-time dithering, or if preprocessing is always preferable
   - Recommendation: Defer dithering to Phase 2.4 or later. If needed, start with canvas-dither (npm package, easier integration) and test performance before committing

2. **IBM Plex Mono Variable Font Availability**
   - What we know: IBM Plex Sans has a variable font version; IBM Plex Mono available through next/font/google with specific weights
   - What's unclear: Whether a variable font version exists for IBM Plex Mono (search results inconclusive)
   - Recommendation: Use standard weights (400, 500, 600) via next/font/google. Variable font would save bytes but isn't critical for three weights

3. **Optimal Text-Shadow Layer Count**
   - What we know: Examples range from 3-7 shadow layers; more layers = more glow depth but higher rendering cost
   - What's unclear: Exact performance threshold on target devices (mobile, tablets)
   - Recommendation: Start with 5 layers (3 white + 2 colored) for subtle glow, 7 layers for prominent headings. Test on target devices and adjust if needed

4. **Box-Drawing Character Cross-Browser Rendering**
   - What we know: Unicode box-drawing characters (U+2500–U+257F) are standard; rendering quality depends on font
   - What's unclear: Whether IBM Plex Mono renders box-drawing characters well on all platforms
   - Recommendation: Test protheus components with IBM Plex Mono. If rendering is poor, consider fallback to Courier New for box-drawing elements only
